# Documentation Workspace Guidance

## Purpose
The `docs/` workspace is the canonical knowledge base for the Diamond language, runtime, and ecosystem. Every addition or update must reinforce the core commitments described in the specification trilogy (`diamond.md`, `diamond2.md`, `diamond3.md`):

1. **Secure-by-default execution** — articulate how documents uphold capability discipline, sandboxing, and zero ambient authority.
2. **Resumable effects** — capture continuation semantics, effect lifecycles, and durable workflows.
3. **Semantic intent capture** — ensure explanations map to the language’s probabilistic decision model and structural typing guarantees.

All documentation must remain synchronized with the codebase, roadmap, and governance process.

## Content Lifecycle

1. **Draft**  
   - Open an issue describing scope, spec touchpoints, and intended audience.  
   - Create an outline referencing relevant sections of the trilogy and active RFCs.

2. **Review**  
   - Route drafts through the appropriate working group(s).  
   - Capture feedback in `docs/design-decisions/records/` when it alters direction or clarifies rationale.

3. **Publish**  
   - Merge documentation with versioned changelog entries (`CHANGELOG.md`).  
   - Update cross-references (link to code, benchmarks, scripts).  
   - Announce updates in the project communication channel with a summary and affected components.

4. **Maintain**  
   - Schedule quarterly audits per directory; verify accuracy against current implementations.  
   - Mark stale content with `> **Status:** Outdated` and create follow-up issues.  
   - Archive superseded documents under `docs/**/archive/` with a deprecation note and replacement links.

## Authoring Standards

- **Front Matter**: Every major document starts with metadata.  
  ```/dev/null/docs_front_matter.md#L1-8
  Title: <document title>
  Authors: <maintainers>
  Version: <Semantic version or date>
  Status: {Draft|Active|Deprecated}
  Spec Alignment: <diamond.md §#, diamond2.md §#, diamond3.md §#>
  Last Updated: <ISO-8601 date>
  Summary: <2-3 sentence overview>
  ```
- **Voice & Tone**: Clear, technical, and actionable. Avoid marketing prose.
- **Structure**: Use hierarchical headings (`#`, `##`, `###`). Table of contents for documents >2 levels deep.
- **Cross-linking**: Link to related spec sections, RFCs, benchmarks, and code paths.
- **Accessibility**: Prefer text-based diagrams (Mermaid/PlantUML). Provide alt text for images.
- **Examples**: Reference real `.dm` snippets stored in `examples/`; ensure they compile once the toolchain is operational.
- **Change Tracking**: Summaries of substantive edits belong in `CHANGELOG.md` and document-specific “History” sections.

---

## Governance & Compliance

- **Spec Edits**: Any change to `docs/spec/` requires an approved RFC and sign-off from Language + Runtime leads.
- **Security Material**: Updates to `docs/security/` must be reviewed by the Security WG and include threat impact analysis.
- **Bootstrapping Docs**: Align with data governance policies, including privacy constraints and synthetic corpus audit logs.
- **Release Evidence**: Roadmap milestones and acceptance criteria go in `docs/spec/roadmap.dm`; ensure supporting docs link to the relevant milestones.
- **Telemetry Backlinks**: When documentation references metrics or benchmarks, cite the originating report in `benchmarks/reports/`.

---

## Quality Checklist

Before merging documentation:

- [ ] References to `diamond.md`, `diamond2.md`, and `diamond3.md` remain accurate.
- [ ] Linked RFCs or issues exist and are open/merged as appropriate.
- [ ] Examples and pseudocode reflect current syntax (square-bracket generics, explicit `perform`, module capability requirements).
- [ ] Security implications are noted, especially for capability or effect changes.
- [ ] Resumability and continuation considerations are described where applicable.
- [ ] Telemetry/metrics expectations are defined, or explicitly marked TBD.
- [ ] Metadata and changelog entries are updated.

---

## Contribution Etiquette

- Use relative links (`../spec/overview.md`) to enable local browsing.
- Avoid duplicating information; reference authoritative sections instead.
- Annotate TODOs with owner and due date, e.g., `TODO(ml-wg, 2025-02-01): Flesh out verifier API`.
- Store large assets (videos, binaries) externally; link with access notes.
- Encourage translation/localization but keep English canonical until a localization process exists.
