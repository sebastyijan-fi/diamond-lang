# HIR Crate Guidance

## Purpose
The `hir/` crate implements the High-level Intermediate Representation for the Diamond compiler. HIR bridges the gap between the syntactic AST produced by the frontend and the low-level Wasm code generated by the backend. This layer is responsible for semantic analysis, type inference, effect inference, and capability validation—ensuring that Diamond code is not only syntactically valid but semantically sound and security-compliant.

All implementations must align with the foundational manuscripts:

- **`diamond.md`** — Intent-oriented design, decision semantics (`<>`), and capability discipline.
- **`diamond2.md`** — Architectural feasibility, synthetic bootstrapping strategy, and zero-trust WebAssembly runtime.
- **`diamond3.md`** — Grammar specification, semantic typing, algebraic effects, and module-level capability injection rules.

---

## Scope & Responsibilities

### Core Transformations
1. **AST to HIR Lowering**: Transform the untyped AST into a typed, resolved HIR.
2. **Name Resolution**: Resolve identifiers to declarations, handle imports, manage scopes.
3. **Type Inference**: Implement bidirectional type inference with support for semantic refinements.
4. **Effect Inference**: Track algebraic effects through the program, validate handler coverage.
5. **Capability Validation**: Ensure capability requirements are explicit and correctly propagated.

### Key Data Structures
- **HIR Nodes**: Typed representations of expressions, statements, declarations, and modules.
- **Type Environment**: Maps identifiers to their inferred or declared types.
- **Effect Context**: Tracks active effect handlers and required capabilities.
- **Symbol Table**: Centralized registry of all named entities with their metadata.
- **Diagnostics**: Rich error and warning collection with source span attachment.

## Type System Implementation

### Bidirectional Type Inference
The HIR crate implements bidirectional type inference following the pattern:

1. **Synthesis**: Infer the type of an expression from its structure.
2. **Checking**: Verify an expression has an expected type.
3. **Subsumption**: Handle subtyping relationships, especially for semantic refinements.

### Semantic Refinements
Per `diamond3.md`, types can have semantic refinements:

```diamond
type Email = String where { valid_email }
type PositiveInt = Int where { > 0 }
```

The HIR must:
- Parse and represent refinement predicates.
- Track refinement constraints through inference.
- Generate verification obligations for the runtime/verifier.

### Type Representation
```rust
pub enum Type {
    // Primitive types
    Unit,
    Bool,
    Int(IntWidth),
    Float(FloatWidth),
    String,
    
    // Composite types
    Tuple(Vec<Type>),
    Record(Vec<(Symbol, Type)>),
    Variant(Vec<(Symbol, Option<Type>)>),
    Array(Box<Type>),
    
    // Function types (with effects)
    Function {
        params: Vec<Type>,
        result: Box<Type>,
        effects: EffectSet,
    },
    
    // Generic types
    TypeVar(TypeVarId),
    Applied(Box<Type>, Vec<Type>),
    
    // Semantic refinements
    Refined {
        base: Box<Type>,
        predicate: RefinementPredicate,
    },
    
    // Special types
    Never,
    Unknown,
    Error,
}
```

## Effect System Implementation

### Effect Tracking
Every function in Diamond may perform effects. The HIR must:

1. **Infer Effects**: Determine which effects a function body performs.
2. **Validate Handlers**: Ensure performed effects are handled at call sites.
3. **Track Resumability**: Mark which effect handlers support resumption.

### Effect Representation
```rust
pub struct EffectSet {
    pub effects: HashSet<EffectId>,
}

pub struct EffectDecl {
    pub name: Symbol,
    pub operations: Vec<EffectOp>,
}

pub struct EffectOp {
    pub name: Symbol,
    pub param_types: Vec<Type>,
    pub result_type: Type,
    pub resumable: bool,
}
```

### Effect Inference Rules
- `perform` expressions introduce the effect into the current context.
- Effect handlers eliminate effects from the enclosed scope.
- Functions must declare all effects they may perform (inferred or explicit).
- Unhandled effects at module boundaries are capability violations.

## Capability Validation

### Capability Model
Per `diamond.md` and `diamond3.md`, Diamond uses object-capability security:

```diamond
import std/net requires { Network }
import std/fs requires { FileSystem }
```

The HIR must:
1. **Track Capability Requirements**: Every import with `requires` creates a capability dependency.
2. **Propagate Requirements**: Function types include their capability needs.
3. **Validate at Boundaries**: Module entry points must receive all required capabilities.
4. **Reject Ambient Authority**: No capability can be assumed without explicit grant.

### Capability Representation
```rust
pub struct CapabilitySet {
    pub capabilities: HashSet<CapabilityId>,
}

pub struct CapabilityDecl {
    pub name: Symbol,
    pub description: String,
    pub operations: Vec<Symbol>,
}
```

## Decision Operator Analysis

The Diamond operator `<>` requires special semantic handling:

```diamond
let result = <> {
    route_a -> handle_a()
    route_b -> handle_b()
    _ -> default_handler()
}
```

### Semantic Requirements
1. **Branch Exhaustiveness**: Validate that branches cover the semantic space or have a default.
2. **Type Consistency**: All branches must produce compatible types.
3. **Effect Unification**: Merge effects from all branches.
4. **Routing Metadata**: Preserve semantic routing information for runtime.

## Diagnostic Standards

All diagnostics must be:

1. **Precise**: Include exact source spans for primary and secondary locations.
2. **Actionable**: Suggest specific fixes when possible.
3. **Educational**: Explain why something is an error, referencing spec sections.
4. **Machine-Readable**: Use stable error codes for tooling integration.

### Error Code Ranges
- `E2000–E2099`: Basic type errors (mismatches, undefined types)
- `E2100–E2199`: Generic/polymorphism errors
- `E2200–E2299`: Semantic refinement errors
- `E2300–E2399`: Effect errors
- `E2400–E2499`: Capability errors
- `E2500–E2599`: Decision operator errors
- `E2600–E2699`: Module/import errors

### Diagnostic Example
```rust
Diagnostic::error(E2001)
    .with_message("type mismatch")
    .with_span(expr.span)
    .with_label(expected_span, format!("expected `{}`", expected))
    .with_label(found_span, format!("found `{}`", found))
    .with_note("types must match exactly; consider adding a conversion")
    .with_help("see diamond3.md §4.3 for type compatibility rules")
```

## Public API

### Core Functions
```rust
/// Lower an AST module to HIR, performing all semantic analysis.
pub fn lower_module(ast: &ast::Module) -> Result<HirModule, Vec<Diagnostic>>;

/// Type-check an already-lowered HIR module.
pub fn type_check(hir: &mut HirModule) -> Result<(), Vec<Diagnostic>>;

/// Infer effects for all functions in a module.
pub fn infer_effects(hir: &mut HirModule) -> Result<(), Vec<Diagnostic>>;

/// Validate capability requirements and propagation.
pub fn validate_capabilities(hir: &HirModule) -> Result<CapabilityManifest, Vec<Diagnostic>>;

/// Perform all semantic analysis passes in order.
pub fn analyze(ast: &ast::Module) -> Result<(HirModule, CapabilityManifest), Vec<Diagnostic>>;
```

### Query Interface (Future)
For IDE integration, consider a query-based interface:

```rust
pub fn type_at(hir: &HirModule, position: Position) -> Option<Type>;
pub fn effects_of(hir: &HirModule, fn_id: FnId) -> EffectSet;
pub fn capabilities_of(hir: &HirModule, module_id: ModuleId) -> CapabilitySet;
```

## Testing Strategy

### Unit Tests
- Type unification correctness.
- Effect inference for isolated expressions.
- Scope and name resolution.

### Integration Tests
- Full module type checking.
- Cross-module capability propagation.
- Error recovery and diagnostic quality.

### Golden Tests
- Snapshot HIR dumps for complex programs.
- Diagnostic output for known error cases.
- Capability manifest generation.

### Property Tests
- Type inference is deterministic.
- Effect inference is sound (no untracked effects).
- Capability validation is complete (no ambient authority).

## Coding Standards

1. **Immutable by Default**: Prefer immutable data structures; use `&mut` only when necessary.
2. **Visitor Pattern**: Use visitors for HIR traversal and transformation.
3. **Arenas for Allocation**: Use typed arenas for HIR nodes to improve cache locality.
4. **Interning**: Intern symbols, types, and spans to reduce memory and enable cheap equality.
